📘 Section 1 — What is the Coin Change Problem?

Problem Statement

You are given a list of coin denominations (for example, coins worth 1, 3, and 4 units) and a target amount (for example, 6). Your goal is to find the minimum number of coins from the list that add up to exactly the target amount. You may use each denomination as many times as you want.

Formal definition: Given coins[] = [c1, c2, ..., cn] and integer amount W, find the smallest integer k such that some combination of k coins from the list sums to W. If no such combination exists, return "Impossible".

Worked Example — Step by Step:

Coins = [1, 3, 4], Amount = 6

Let's list all possible combinations that sum to 6:
• 1+1+1+1+1+1 = 6 coins
• 4+1+1 = 3 coins
• 3+1+1+1 = 4 coins
• 3+3 = 2 coins ← This is the minimum!

The answer is 2 coins (two 3-unit coins).

Why is This Problem Important?

The Coin Change Problem is one of the most famous problems in computer science because it perfectly illustrates two key ideas: Optimal Substructure and Overlapping Subproblems. These two properties are what make a problem solvable by Dynamic Programming. The problem also appears in real life — ATM machines, vending machines, and currency exchange systems all need to solve a version of this problem.

More importantly for this course, it is the classic example that proves Greedy algorithms are not always correct. Even though Greedy is faster, it gives the wrong answer for certain coin sets. DP always gives the correct answer.

⚙️ Section 2 — How Dynamic Programming Solves It

The Core Idea: Build Up from Small Answers

Dynamic Programming solves the Coin Change Problem by breaking it into smaller subproblems. Instead of asking "how do I make 11?", we first ask "how do I make 1?", then "how do I make 2?", and so on, all the way up to 11. Each answer is stored in a table so we never compute the same thing twice.

We create an array dp[] where dp[i] = the minimum number of coins to make amount i. We fill this array from left to right, and every time we compute dp[i], we look back at previously computed values to find the best option.

The DP Recurrence Formula

dp[i] = min( dp[i - c] + 1 ) for all coins c where c <= i

In plain English: "The minimum coins to make amount i equals one coin (the coin we just used) plus the minimum coins to make the remainder i - c."

Full Worked Example — Trace Through the Table

Let's trace the algorithm for Coins = [1, 3, 4], Amount = 6:

0Base case0— (none needed)
1dp[0]+1 = 13 > 1, skip4 > 1, skip1coin 1
2dp[1]+1 = 23 > 2, skip4 > 2, skip2coin 1
3dp[2]+1 = 3dp[0]+1 = 14 > 3, skip1coin 3
4dp[3]+1 = 2dp[1]+1 = 2dp[0]+1 = 11coin 4
5dp[4]+1 = 2dp[2]+1 = 3dp[1]+1 = 22coin 1 (or 4)
6dp[5]+1 = 3dp[3]+1 = 2dp[2]+1 = 32coin 3

Final answer: dp[6] = 2. The optimal solution is two 3-unit coins.

The Two Key Properties That Make DP Work

1. Optimal Substructure:
The optimal solution to a big problem contains optimal solutions to its sub-problems. If the best way to make 6 uses a 3-coin, then the best way to make the remaining 3 must also be optimal. If it were not, we could replace it with a better solution and get a better answer for 6 — contradicting our assumption.

2. Overlapping Subproblems:
The same sub-amounts are needed by many different larger amounts. For example, dp[2] is used when computing dp[3], dp[5], and dp[6]. DP computes dp[2] only once and reuses it, unlike naive recursion which recomputes it every time.

Step-by-Step Algorithm

  1. Sort the coins in ascending order (this enables the early-exit optimization).
  2. Create the dp array of size amount + 1 and fill it with infinity (meaning "no solution found yet").
  3. Set dp[0] = 0 — the base case: you need zero coins to make zero amount.
  4. Outer loop: For each amount i from 1 to the target amount, do the following:
  5. Inner loop: For each coin c in the sorted coins list:
  6. If c > i, break the inner loop (coin is too large; all remaining coins are also too large because the list is sorted).
  7. If dp[i - c] + 1 < dp[i], update dp[i] with the better value.
  8. Return dp[amount] as the final answer. If it is still infinity, the answer is "Impossible".
🌳 Section 3 — The Recursion Tree: Why Naive Recursion Fails

What is Naive Recursion?

A naive recursive approach would try every possible combination of coins by branching out at each step. For each amount, it tries subtracting every coin and recursively solves the smaller amount. This is correct but extremely slow.

The Problem — Exponential Time:

For Coins = [1, 3, 4] and Amount = 6, the naive recursion tree has dozens of nodes. Many of those nodes represent the same amount being solved multiple times. For example, the subproblem "how do I make 2?" might appear in 5 different branches of the tree.

As the amount grows, the number of repeated computations grows exponentially. The time complexity of naive recursion is O(n^W) — "n to the power of W" — which becomes impossibly slow for even moderate inputs (e.g., amount = 30 with 3 coins = 3^30 = over 200 billion operations).

How to Read the Recursion Tree (in This Tool)

  • Purple nodes — a recursive call to solve a sub-amount. The number inside is the amount being solved.
  • Green nodes — base case. Amount = 0, so 0 more coins are needed. Recursion stops here.
  • Pink nodes with orange dashes — this amount has already appeared elsewhere in the tree. This is an overlapping subproblem. In naive recursion, this would be recomputed from scratch. DP stores the answer the first time and reuses it.
  • Edge labels — the coin that was subtracted to go from the parent node to the child node.

How DP Eliminates the Redundancy

Dynamic Programming solves each sub-amount exactly once and stores the result in the dp[] array. When the same sub-amount is needed again (from a different branch), DP simply looks up the stored answer in O(1) time instead of recomputing it. This transforms the algorithm from exponential O(n^W) time to polynomial O(n · W) time.

Key Takeaway:
The Recursion Tree is a visual proof of why DP is needed. Every orange-dashed node in the tree represents wasted computation that DP avoids. The more orange nodes you see, the bigger the speedup DP provides over naive recursion.
📊 Section 4 — Complexity Analysis (CLO-1)

Time Complexity: O(n · W)

The DP algorithm has two nested loops. The outer loop runs W times (once for each amount from 1 to the target). The inner loop runs at most n times (once for each coin). Therefore, the total number of operations is at most n × W, giving a time complexity of O(n · W).

Where n = number of coin denominations and W = target amount. For example, with 3 coins and amount 6, the theoretical maximum is 3 × 6 = 18 operations.

Space Complexity: O(W)

The algorithm only needs to store the dp[] array, which has W + 1 entries (one for each amount from 0 to W). No other significant memory is used. Therefore the space complexity is O(W).

Sorting Optimization — CLO-2 (Comparisons Saved)

Before the main loops, the coins are sorted in ascending order. This enables an important optimization: when the inner loop encounters a coin c that is larger than the current amount i, it immediately breaks out of the inner loop. Because the coins are sorted, every remaining coin is also larger than i, so there is no point checking them.

This means the actual number of operations is often significantly less than the theoretical maximum of n × W. The Comparisons Saved counter in the tool tracks exactly how many inner-loop iterations were skipped because of this sorting optimization. The Growth Chart visualizes the difference between actual and theoretical operations.

⚔️ Section 5 — Greedy vs Dynamic Programming (CLO-3)

What is the Greedy Algorithm?

The Greedy algorithm for Coin Change works as follows: sort the coins in descending order (largest first), then repeatedly pick the largest coin that fits into the remaining amount. It never backtracks or reconsiders a decision.

Greedy is very fast — it runs in O(n log n) time for sorting plus O(W/c_min) for the loop. But speed comes at a cost: it is not always correct.

Classic Counterexample — Greedy Fails:

Coins = [1, 3, 4], Amount = 6

Greedy approach:
• Largest coin that fits: 4. Remaining: 6 − 4 = 2.
• Largest coin that fits: 1. Remaining: 2 − 1 = 1.
• Largest coin that fits: 1. Remaining: 1 − 1 = 0.
• Result: 3 coins (4 + 1 + 1)

DP approach:
• Tries all combinations. Finds that 3 + 3 = 6.
• Result: 2 coins (3 + 3) ← This is the true minimum!

Greedy committed to the 4-coin too early and could not go back. DP explored all options and found the true optimum.

When Does Greedy Work?

Greedy works correctly only for coin systems that satisfy the Greedy Property — meaning the locally optimal choice at every step is also globally optimal. The standard US coin system [1, 5, 10, 25] satisfies this property, so Greedy always gives the correct answer for US coins. You can verify this with the "US Coins (41)" preset — both Greedy and DP will give the same answer.

Why DP is Provably Correct — Optimal Substructure Proof

DP is correct because the Coin Change Problem has Optimal Substructure. Here is the proof by contradiction:

Assume the optimal solution to amount W uses coin c. Then the remaining amount W − c must also be solved optimally. Why? Suppose it were not optimal — suppose there is a better solution for W − c using fewer coins. Then we could replace our sub-solution with that better one, giving us a better solution for W overall. But this contradicts our assumption that we already had the optimal solution for W. Therefore, every sub-solution in the optimal answer must itself be optimal. This is exactly what DP guarantees.

Common Mistakes to Avoid

  • Forgetting the base case: If you do not set dp[0] = 0, the entire table will stay at infinity because every recurrence depends on dp[0].
  • Using infinity in arithmetic: Before using dp[i - c], always check that it is not infinity. Otherwise infinity + 1 causes an overflow error.
  • Confusing n and W: n is the number of coin types (the length of the coins array). W is the target amount. The complexity is O(n · W), not O(n^2).
  • Thinking Greedy always works: It does not. Always test with the [1, 3, 4] → 6 counterexample before trusting a Greedy solution.

When Should You Use Dynamic Programming?

Use Dynamic Programming when a problem has both of these properties:

  • Optimal Substructure: The optimal solution to the full problem can be built from optimal solutions to its sub-problems.
  • Overlapping Subproblems: The same sub-problems are solved multiple times in a naive recursive approach.

The Coin Change Problem has both properties, which is why DP is the correct approach. Other classic DP problems include: Longest Common Subsequence, 0/1 Knapsack, Matrix Chain Multiplication, and Fibonacci (with memoization).
