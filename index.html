📘 Coin Change Problem — Overview

Problem Definition

Given a set of coin denominations coins[] and a target amount, find the minimum number of coins needed to make exactly that amount. You may use each coin denomination an unlimited number of times.

Input: A list of coin values (e.g., [1, 3, 4]) and a target integer (e.g., 6).
Output: The minimum number of coins, or "Impossible" if no combination can reach the target.

Worked Example: Coins = [1, 3, 4], Amount = 6
Possible combinations: 1+1+1+1+1+1 (6 coins), 4+1+1 (3 coins), 3+3 (2 coins).
Optimal answer: 2 coins (using two 3-cent coins).

Why This Problem Matters

The Coin Change Problem is a canonical example of Dynamic Programming. It appears in real-world systems such as ATM cash dispensing, currency exchange, and resource allocation. It is also the classic counterexample that proves Greedy algorithms are not always optimal for arbitrary coin sets.

⚙️ Dynamic Programming — How It Works

Core Idea

Instead of solving the big problem directly, DP breaks it into smaller overlapping subproblems. We build a table dp[] where dp[i] stores the minimum coins needed to make amount i. We solve dp[1], then dp[2], all the way up to dp[amount], reusing previously computed answers.

How It Works — Step by Step

  1. Sort the coins in ascending order (enables early-exit optimization).
  2. Create array dp[0..amount], fill everything with infinity (unknown).
  3. Set dp[0] = 0 — base case: zero coins needed for zero amount.
  4. For each amount i from 1 to target, try every coin c.
  5. If c > i, break the inner loop (coin is too large — sorting benefit).
  6. If dp[i−c] + 1 < dp[i], update dp[i] with the better value.
  7. After all loops, dp[amount] holds the optimal answer.
Key Insight — Optimal Substructure:
The optimal solution to amount i contains the optimal solution to amount i−c. This is what makes DP correct: every sub-answer we store is itself optimal, so building on it guarantees the final answer is also optimal.

Overlapping Subproblems

A naive recursive approach would recompute the same sub-amounts many times (see the Recursion Tree above). For example, computing the answer for amount 4 might require computing amount 2 multiple times across different branches. DP stores each answer exactly once, reducing exponential time to polynomial time.

📊 Complexity Analysis

Time Complexity

The algorithm has two nested loops: the outer loop runs W times (the target amount) and the inner loop runs at most n times (the number of coins). Therefore:

Time: O(n · W)  |  Space: O(W)

Where n = number of coin denominations and W = target amount.

Complexity Breakdown

Sorting Optimization (CLO-2)

By sorting coins in ascending order before the loops, we can break early from the inner loop the moment a coin exceeds the current amount. Since all remaining coins are also larger (sorted order), there is no point checking them. This reduces the actual number of operations below the theoretical maximum of n × W, as shown by the Comparisons Saved counter in the tool above.

⚔️ Greedy vs Dynamic Programming (CLO-3)

What is the Greedy Approach?

The Greedy algorithm sorts coins in descending order and always picks the largest coin that fits into the remaining amount. It is fast and simple, but it is not always correct.

Counterexample — Greedy Fails:
Coins = [1, 3, 4], Amount = 6

Greedy: Picks 4 (remaining: 2) → picks 1 (remaining: 1) → picks 1 (remaining: 0). Result: 3 coins.
DP: Finds 3 + 3 = 6. Result: 2 coins.

Greedy committed to the 4-coin too early and could not backtrack. DP explored all possibilities and found the true optimum.

When Does Greedy Work?

Greedy works correctly for coin systems that satisfy the Greedy Property — where the locally optimal choice is always globally optimal. The standard US coin set [1, 5, 10, 25] satisfies this property, so Greedy always gives the correct answer for US coins. However, for arbitrary coin sets (like [1, 3, 4]), this property does not hold, and DP is required.

Why DP is Correct — Optimal Substructure Proof

DP is correct because the Coin Change Problem has Optimal Substructure: the optimal solution to amount W contains optimal solutions to smaller amounts. Formally: if the optimal solution to W uses coin c, then the remaining W−c must also be solved optimally. If it were not, we could replace that sub-solution with a better one, contradicting the assumption that our solution for W was optimal.

🔍 Edge Cases & When to Use DP

Edge Cases

  • Amount = 0: Always returns 0 coins. The base case dp[0] = 0 handles this directly.
  • Impossible case: If no combination of coins can reach the amount (e.g., coins = [2], amount = 3), dp[amount] remains infinity and the tool reports "Impossible".
  • Single coin = amount: Returns 1 coin immediately.
  • Large amounts: DP handles large amounts efficiently in O(n·W) time, unlike exponential recursion.

When to Use Dynamic Programming

  • When the problem has Optimal Substructure (optimal solution built from optimal sub-solutions).
  • When the problem has Overlapping Subproblems (same sub-problems are solved multiple times in recursion).
  • When a Greedy approach does not guarantee correctness for all inputs.
  • When the input size is manageable (polynomial in W and n).
